# ARM64 Backend ‚Äî Design Spec

## Overview

Add full AArch64 (ARM64) support to KernRift across three platforms: Linux (`aarch64-sysv`), macOS (`aarch64-macho`), and Windows (`aarch64-win`). The default `kernriftc` output becomes a fat binary (`KRBOFAT`) containing both x86_64 and ARM64 native code slices, compressed with LZ4. A new `--arch` flag allows targeting a single architecture. Scope: `crates/krir`, `crates/emit`, `crates/kernriftc`, `crates/kernrift`, `docs/spec`.

---

## 1. Phases

### Phase 1 ‚Äî AArch64 Target Contracts

Define the three ABI target models in spec docs and in `crates/krir/src/lib.rs`:

- `aarch64-sysv` ‚Äî Linux, AAPCS64
- `aarch64-macho` ‚Äî macOS Apple Silicon, AAPCS64 + Apple stack rules
- `aarch64-win` ‚Äî Windows ARM64

Each target gets a `BackendTargetContract` struct in `krir/src/lib.rs`. Because the existing register fields are typed to `X86_64IntegerRegister`, Phase 1 introduces a register abstraction:

- A new `AArch64IntegerRegister` enum (`X0`‚Äì`X30`, `Sp`, `Xzr`)
- A new `IntegerRegister` enum wrapping both: `IntegerRegister::X86_64(X86_64IntegerRegister)` and `IntegerRegister::AArch64(AArch64IntegerRegister)`
- All `BackendTargetContract` register fields are updated from `Vec<X86_64IntegerRegister>` to `Vec<IntegerRegister>` (or the appropriate scalar type)

A new `TargetArch::AArch64` variant is added alongside the existing `TargetArch::X86_64`.

`validate()` has two guards that both require changes:

1. **`known_combo` match** (lines 3181‚Äì3196 in `krir/src/lib.rs`): currently exhaustively lists three `(BackendTargetId, TargetArch, TargetAbi)` tuples for x86_64. Three new arms must be added:
   - `(Aarch64Sysv, AArch64, Aapcs64)`
   - `(Aarch64MachO, AArch64, Aapcs64)`
   - `(Aarch64Win, AArch64, Aapcs64Win)`

2. **Standalone arch guard** (line 3203): `if self.arch != TargetArch::X86_64 { return Err(...) }` ‚Äî this guard is now redundant once the `known_combo` match covers AArch64 and can be removed.

The `IntegerRegister` wrapper enum must derive `PartialOrd` and `Ord` (in addition to `Debug, Clone, Copy, PartialEq, Eq`) because `validate()` collects register lists into `BTreeSet<_>`. Without `Ord`, the `BTreeSet` construction fails to compile.

`FutureScalarReturnConvention::registers()` currently returns `&'static [X86_64IntegerRegister]` and is used in a `validate()` loop that will become `BTreeSet<IntegerRegister>` after migration. This is a type mismatch. `FutureScalarReturnConvention` must be extended to either accept `IntegerRegister` variants directly or have a parallel `registers_abstract() -> &'static [IntegerRegister]` method. The simplest approach: add `IntegerRegister` arms to the existing `registers()` return type by changing the return type to `Vec<IntegerRegister>` and wrapping all existing `X86_64IntegerRegister` values.

The `validate()` stack alignment check (`stack_alignment_bytes != 16 ‚Üí error`) is preserved as-is. All three AArch64 targets use `stack_alignment_bytes = 16`. The Windows ARM64 leniency for 8-byte alignment at leaf calls is a codegen-level detail, not a contract-level property.

`instruction_pointer` field: all three AArch64 contracts set `instruction_pointer: "pc"` (vs `"rip"` for x86_64).

`X18` register exclusion: `X18` is excluded from `integer_registers` for both `aarch64-sysv` (conservative, mirrors the Linux kernel ABI where `x18` is shadow-call-stack reserved) and `aarch64-win` (TEB pointer, hardware-reserved).

No instruction emission in this phase. The deliverable is: `BackendTargetContract::aarch64_sysv()`, `aarch64_macho()`, `aarch64_win()` compile cleanly and pass `validate()`.

### Phase 2 ‚Äî Instruction Selection + Register Allocation (Linux)

Add AArch64 instruction selection and register allocation for `aarch64-sysv`. Covers the same executable KRIR subset that x86_64 supports today: scalar ops, loads/stores, function calls, tail calls, MMIO. macOS pointer authentication (`PACIA`/`AUTIA`) is deferred to Phase 4.

**Deliverable:** `kernriftc --arch arm64 <file.kr>` produces a single-arch arm64 krbo. The `--arch` flag is introduced in Phase 6 for the full fat-binary CLI, but the single-arch arm64 output path is wired in Phase 2 to make the deliverable concrete and testable via the existing test harness.

### Phase 3 ‚Äî Assembly + ELF Emission (Linux)

`--emit=asm` and `--emit=elfobj` for `aarch64-sysv`. Adds:
- `docs/spec/aarch64-asm-linear-subset-v0.1.md`
- `docs/spec/aarch64-object-linear-subset-v0.1.md`

### Phase 4 ‚Äî macOS ARM64

Full stack for `aarch64-macho`: codegen (including pointer authentication), Mach-O object emission, assembly.

### Phase 5 ‚Äî Windows ARM64

Full stack for `aarch64-win`: codegen, PE/COFF emission, assembly.

### Phase 6 ‚Äî Fat Binary Container + CLI

Introduce the `KRBOFAT` container format, full `--arch` flag (including the dual-file output logic for non-fat emits), and fat binary support in `kernrift`. Default `kernriftc <file.kr>` becomes a fat binary.

## 2. Fat Binary Container Format (`KRBOFAT`)

### Design principle

Single-arch krbo files remain byte-identical to the current format. The fat container is a new outer wrapper. A file is either a plain krbo or a `KRBOFAT` ‚Äî distinguished by the first 8 bytes. The existing single-arch krbo magic is `KRBO` (4 bytes, `bytes[0..4]`). The fat magic is `"KRBOFAT\0"` (8 bytes). Since the fat magic begins with `KRBO`, **fat detection must be performed first** by checking all 8 bytes before falling back to the 4-byte single-arch check. Single-arch detection must never be attempted on a fat binary.

### File layout

```
[0..8]    magic:      "KRBOFAT\0"   (7 ASCII bytes + null terminator = 8 bytes total)
[8..12]   version:    u32 = 1       (little-endian throughout)
[12..16]  arch_count: u32
[16..]    entries:    arch_count √ó ArchEntry (32 bytes each)
          padding:    0-pad to next 16-byte boundary from file start
          slices:     arch_count √ó LZ4-compressed krbo bytes
```

### `ArchEntry` (32 bytes)

```
arch_id:            u32   (1 = x86_64, 2 = arm64)
compression:        u32   (0 = none, 1 = lz4)
offset:             u64   (byte offset from file start to compressed slice)
compressed_size:    u64   (byte length of stored slice)
uncompressed_size:  u64   (byte length after decompression)
```

### Architecture IDs

| ID | Architecture |
|----|-------------|
| 1  | `x86_64`    |
| 2  | `arm64`     |

These IDs intentionally match the existing `KRBO_ARCH_X86_64 = 0x01` constant in the single-arch krbo header so that the numbering is consistent across both formats. Reserved: IDs 3+ for future targets (RISC-V, etc.).

### Compression

Each slice is compressed independently with LZ4 (frame format). `compression = 0` is valid and means the slice is stored uncompressed. Default for fat binaries: `compression = 1` (LZ4).

LZ4 frame format includes a content checksum field. `lz4_flex` writes and verifies this checksum by default. Slice integrity is fully delegated to LZ4's own frame checksum ‚Äî no additional checksum field in `ArchEntry`.

**Dependency:** `lz4_flex` crate (pure Rust, no C bindings). Added to workspace dependencies.

### Size expectation

LZ4 typically compresses native code to 50‚Äì60% of original size. A fat binary containing both arches is therefore roughly **1.1‚Äì1.2√ó** the size of a single-arch krbo rather than 2√ó.

### Reading in `kernrift`

Detection is fat-first. The 8-byte fat magic check must precede the 4-byte single-arch magic check:

```
1. if file[0..8] == "KRBOFAT\0"  ‚Üí  fat path:
       find ArchEntry where arch_id == host_arch
       seek to entry.offset
       lz4-decompress entry.compressed_size bytes ‚Üí entry.uncompressed_size bytes
       load decompressed slice as single-arch krbo
2. elif file[0..4] == KRBO_MAGIC  ‚Üí  single-arch path (unchanged)
3. else                            ‚Üí  error: unrecognised format
```

If the fat binary does not contain a slice for the host architecture:
```
error: <file> does not contain a slice for arm64
```

## 3. CLI Design

### `--arch` vs `--target`

The existing `--target <triple>` flag (e.g. `--target aarch64-sysv`) selects a full target triple in the backend-emit path. The new `--arch x86_64|arm64` flag is a simplified top-level flag for the primary compile path. They serve different layers:

- `--target` remains as-is for backend-emit triple selection (`--emit=elfobj --target aarch64-sysv`)
- `--arch` is a new top-level flag for fat/single-arch krbo control
- When both are present, `--arch` controls fat vs single-arch output; `--target` specifies the full triple for the emit target
- For the default compile path (no `--emit`), `--arch` is the only flag needed ‚Äî `--target` is not required

### `--arch` flag

Valid values: `x86_64`, `arm64` (also accept `aarch64` as alias for `arm64`).

```
kernriftc <file.kr>                               # fat krbo (default)
kernriftc --arch x86_64 <file.kr>                 # single-arch krbo, x86_64
kernriftc --arch arm64  <file.kr>                 # single-arch krbo, arm64
```

### Non-fat emit targets

`--emit=elfobj`, `--emit=asm`, and `--emit=staticlib` produce inherently single-arch output. Behaviour:

- **Without `--arch`**: produce two files (one per arch), print a message.
- **With `--arch`**: produce one file (stem unchanged).

```
kernriftc --emit=elfobj <file.kr>
  ‚Üí foo-x86_64.o + foo-arm64.o
  ‚Üí emitted foo-x86_64.o and foo-arm64.o (use --arch to target one architecture)

kernriftc --arch arm64 --emit=elfobj <file.kr>
  ‚Üí foo.o
```

Same pattern for `--emit=asm` and `--emit=staticlib`.

### Arch-neutral emit targets

`--emit=krir`, `--emit=lockgraph`, `--emit=caps`, `--emit=contracts` are IR-level outputs ‚Äî `--arch` is ignored for these.

### `kernrift` runner

No `--arch` flag. Always selects the host architecture automatically using the fat-first detection described in Section 2.

## 4. AArch64 Calling Conventions

### `aarch64-sysv` (Linux) and `aarch64-macho` (macOS) ‚Äî AAPCS64

| Role | Registers |
|------|-----------|
| Integer args / return | `x0`‚Äì`x7` (return in `x0`) |
| Indirect result pointer | `x8` |
| Caller-saved (scratch) | `x9`‚Äì`x15` |
| Linker scratch (reserved) | `x16` (IP0), `x17` (IP1) ‚Äî not used by KernRift |
| Platform-reserved | `x18` ‚Äî excluded from `integer_registers` (shadow-call-stack on Linux) |
| Callee-saved | `x19`‚Äì`x28` |
| Frame pointer | `x29` |
| Link register | `x30` |
| Stack pointer | `sp` (16-byte aligned at calls) |
| Instruction pointer | `pc` (used in `BackendTargetContract.instruction_pointer`) |

`x16` and `x17` are explicitly reserved for linker use (branch-island veneers) and must not be used by generated code or as indirect tail-call registers. `x18` is excluded from `integer_registers` for both Linux (shadow-call-stack reserved in the Linux kernel ABI) and Windows (TEB pointer).

macOS-specific addition (Phase 4 only): pointer authentication on return addresses (`PACIA`/`AUTIA`). Scoped out of Phase 2.

### `aarch64-win` (Windows ARM64)

Same integer argument registers (`x0`‚Äì`x7`). Differences from AAPCS64:

- Stack alignment: 8-byte at leaf calls, 16-byte at non-leaf calls. `BackendTargetContract.stack_alignment_bytes = 16` (the conservative value); the leaf-call 8-byte leniency is handled in codegen, not in the contract.
- `x18` is reserved as the TEB (Thread Environment Block) pointer ‚Äî excluded from `integer_registers` (same as Linux, different reason).
- `x16`, `x17` reserved as on Linux/macOS.
- `instruction_pointer: "pc"` (same as other AArch64 targets).

### Tail calls on AArch64

x86_64 uses `jmp` for tail calls. AArch64 equivalent:

- **Direct tail call:** `b target` (branch without link) ‚Äî used when the target is a known intra-module symbol resolvable within the ¬±128 MB branch range.
- **Indirect tail call:** `br x9` (branch to register, using caller-saved scratch `x9`) ‚Äî used for indirect/unknown targets.

`x16` (IP0) and `x17` (IP1) are explicitly excluded as indirect tail-call registers because the linker may clobber them when inserting branch-island stubs. `x9` is safe: it is a caller-saved scratch register that the linker never touches. All KernRift tail calls are intra-module in the current executable subset; cross-object tail calls are not supported.

The tail-call constraint (callee must fit in the same register frame, ‚â§ 6 args under SysV) maps cleanly to AArch64.
